# The Workflow

The ZenAI workflow is a linear sequence of phases. Each phase has a clear purpose, a specific tool, and a clean handoff to the next. The entire cycle runs inside Claude Code (or any AI coding assistant that supports skills and subagents).

## Overview

```
Plan → /clear → Execute → /clear → Verify → Fix → Review → Show & Tell → Archive
```

| Phase | Who | Key Action |
|---|---|---|
| **Plan** | Default agent + skills | Interactive planning → `/dev-docs` |
| **Execute** | Default agent + skills | Work through tasks → `/update-dev-docs` to checkpoint |
| **Verify** | `@zen-audit` + `@zen-review` | Fire-and-forget agents → reports in `handoff/` |
| **Fix** | Default agent + `/fix-plan` | Triage findings → fix docs → execute fixes |
| **Review** | Default agent + `/code-review` | Interactive code review ceremony |
| **Show & Tell** | Default agent + `/show-n-tell` | Final quality gate — demo to stakeholder |

## Phase Details

### 1. Plan

Enter plan mode. Discuss, iterate, research. Load domain skills as needed for specialized knowledge.

After approval, persist the plan:

```
/dev-docs <feature-name>
```

This creates three files in `docs/plan/active/<feature>/`:

| File | Purpose |
|------|---------|
| `plan.md` | Strategy, architecture, approach, risks |
| `context.md` | Key decisions, references, session knowledge |
| `tasks.md` | Checklist: Part → Section → Task hierarchy |

**Before switching to execution:**
```
/update-dev-docs → /clear
```

### 2. Execute

Paste the handoff prompt (generated by `/update-dev-docs`) into the fresh session. Work through tasks with the default agent. Load skills as needed for domain expertise.

Task status markers:

| Marker | Meaning |
|--------|---------|
| `[ ]` | Pending |
| `[~]` | In progress |
| `[x]` | Complete |
| `[!]` | Blocked |

Commit in small increments. Run `/update-dev-docs` to checkpoint progress before context limits, session switches, or topic changes.

**Before switching to verification:**
```
/update-dev-docs → /clear
```

### 3. Verify (Audit + Review)

Run both verification agents in parallel (separate terminals):

```
Terminal 1: @zen-audit "Audit <feature> tasks"
Terminal 2: @zen-review "Review <feature> implementation — reference <domain-skill>"
```

Both agents:
- Run in isolated context windows (no builder bias)
- Read all files in the phase directory
- Write structured reports to `handoff/`
- NEVER modify source code
- Include a verdict: PASS/FAIL with categorized findings (CRITICAL/MAJOR/MINOR)

Report naming: `handoff/<feature>-audit-<MMDD>-<i>.md`, `handoff/<feature>-review-<MMDD>-<i>.md`

### 4. Fix

After both agents finish, triage and fix findings:

```
/fix-plan <feature>
```

This reads all audit and review reports from `handoff/`, deduplicates overlapping findings, and creates fix docs inside the existing phase folder:

| File | Purpose |
|------|---------|
| `audit-review-fix-plan.md` | Triage: fix now / intentional / defer to backlog |
| `audit-review-fix-context.md` | Combined findings summary |
| `audit-review-fix-tasks.md` | Fix checklist |

Enter `/plan` to review the triage, approve, then execute fixes.

### 5. Review

Interactive code review ceremony using the default agent:

```
/code-review
```

Read the zen-review report first (`handoff/<feature>-review-*.md`), then walk through findings interactively. Fix issues found during review.

### 6. Show & Tell

Final quality gate — the engineer demonstrates completed work to the stakeholder:

```
/show-n-tell
```

Read the zen-audit report first (`handoff/<feature>-audit-*.md`). Walk through features, design decisions, test execution, coverage, tech debt, security, and docs sync. One section at a time with pauses for questions.

### 7. Archive

After all phases complete, move the feature from `active/` to `archive/` and clean up:

```bash
mv docs/plan/active/<feature>/ docs/plan/archive/
rm handoff/<feature>-*
```

## The Golden Rule

**Always `/update-dev-docs` before `/clear`. Always paste the handoff prompt after `/clear`.**

This one discipline prevents the biggest productivity killer: losing session knowledge during context resets. The handoff prompt captures insights, corrections, decisions, and current state so the next session doesn't start from zero.

## Multi-Terminal Setup

| Terminal | Purpose |
|---|---|
| **Terminal 1** | Main work — planning, execution, interactive review |
| **Terminal 2** | `@zen-audit` — then fix findings |
| **Terminal 3** | `@zen-review` — or combine with Terminal 2 for fixes |

Verification agents run in parallel. Fix work runs sequentially (avoids merge conflicts on the same files).

## Quick Reference

```
plan mode → /dev-docs → /update-dev-docs → /clear
→ paste handoff prompt → execute tasks → /update-dev-docs → /clear
→ @zen-audit + @zen-review (parallel) → wait
→ /fix-plan → /plan → approve → execute fixes
→ /code-review → /show-n-tell
→ archive → next feature
```

## Cross-Cutting Skills

These can be invoked at any point in the workflow:

| Skill | When | Purpose |
|---|---|---|
| `/update-dev-docs` | Before every `/clear` | Checkpoint progress + generate handoff prompt |
| `/capture` | After breakthroughs | Persist insights into project documentation |
| `/add-backlog` | Anytime | Quick capture of future ideas to `docs/plan/backlog/` |

## Dev-Docs Lifecycle

```
docs/plan/backlog/    → lightweight future items
docs/plan/active/     → current work (3+ files per feature)
docs/plan/archive/    → completed features
handoff/              → ephemeral verification reports (gitignored)
```

Phases accumulate additional docs from iterations. A feature folder might grow beyond the initial 3 files as you iterate on the plan during execution.
